# Fix Plan: Rikugan UI Optimization — Crash Fixes & Performance

## Context

Two issues reported:
1. **Crash when changing API settings in IDA Pro** — no error shown
2. **Code is slow / unoptimized** — Qt polling, threading, and signal anti-patterns throughout the codebase

---

## Phase 1: Fix IDA Pro API Settings Crash

### Root Cause
The crash occurs in `SettingsDialog` when changing providers. The flow:

1. User changes provider in combo → `_on_provider_changed()` is called
2. `_on_provider_changed()` calls `self._config.switch_provider(provider)` which calls `_snapshot_current_provider()`
3. `_snapshot_current_provider()` saves the current provider's `api_key` to `self.providers[name]`
4. **BUT** `_snapshot_current_provider()` does NOT save the `api_key` from `self.provider.api_key` — it saves the value that's already in `self.provider.api_key` (the current provider's key, not the previous provider's key)
5. **More critically**: when `_on_provider_changed()` calls `self._config.switch_provider(provider)` to switch to the new provider, the code path in `RikuganPanelCore._on_settings()` saves the config via `self._config.save(password=dlg.encryption_password)` AFTER the dialog closes. But if the dialog crashes before `_on_accept()` is called, no save happens. However, the crash is likely in the dialog construction or `_on_provider_changed` itself.

### 1.1 Fix `_snapshot_current_provider` — Save the CORRECT key
The bug: when switching from provider A to provider B, `_snapshot_current_provider()` saves provider A's config. But then `switch_provider(B)` overwrites `self.provider.name` BEFORE the snapshot is stored in `self.providers`. Actually looking at the code more carefully, `_snapshot_current_provider()` saves the current provider BEFORE `self.provider.name` is changed, so the name stored is the OLD name. Wait — `_snapshot_current_provider()` saves `name=self.provider.name` which is still the OLD name when called. So it stores:
```python
self.providers[old_name] = {...old values...}
```
Then `self.provider.name = new_name` is set AFTER. This is actually correct.

The REAL issue might be in `_sync_config_from_ui()` called DURING `_on_provider_changed()`. When the user selects a DIFFERENT provider:
1. `_on_provider_changed()` detects `self._config.provider.name != provider` (old != new)
2. It calls `_sync_config_from_ui()` which does `self._config.provider.model = self._get_selected_model_id()` — but `self._get_selected_model_id()` reads from the COMBO that was just cleared/set to the NEW provider's model

Actually, the crash is likely caused by `_sync_config_from_ui()` being called DURING the provider switch, BEFORE the config has been switched. This means `self._config.provider.model` gets overwritten with the OLD provider's UI state applied to the NEW provider slot.

### 1.2 Fix `_on_provider_changed` — Call `_sync_config_from_ui` only when switching AWAY
```python
def _on_provider_changed(self, provider: str) -> None:
    # Save OLD provider settings BEFORE switching
    if self._config.provider.name != provider:
        self._sync_config_from_ui()  # Sync OLD provider's values from UI
    self._config.switch_provider(provider)
    ...
```

Currently the sync call is unconditional when names differ. But the real issue is that `self._sync_config_from_ui()` reads from `self._model_combo.currentText()` which is the NEW provider's model (already updated by Qt). So it's saving the new provider's model into the old provider's saved config slot.

**Fix**: `_sync_config_from_ui()` should snapshot the current (old) provider's values from UI BEFORE `switch_provider()` changes `self.provider.name`. Store the old name, call sync, then switch.

### 1.3 Wrap `_on_provider_changed` in exception handling (APPROVED)
The crash has no error output because there's no try/except. Add:
```python
def _on_provider_changed(self, provider: str) -> None:
    try:
        ...existing logic...
    except Exception as e:
        log_error(f"Provider switch failed: {e}")
        # Restore UI to previous state without re-triggering the signal
        self._provider_combo.blockSignals(True)
        self._provider_combo.setCurrentText(self._config.provider.name)
        self._provider_combo.blockSignals(False)
        self._api_key_edit.setText(self._config.provider.api_key)
        self._api_base_edit.setText(self._config.provider.api_base)
        self._model_combo.setCurrentText(self._config.provider.model)
```

### 1.4 Add `_closed` guard to `_poll_fetcher`
Set `_closed = True` BEFORE stopping timers to prevent race with UI access:
```python
def done(self, result: int) -> None:
    self._closed = True  # BEFORE any timer cleanup
    try:
        self._init_timer.stop()
        self._poll_timer.stop()
        self._fetcher.shutdown()
    finally:
        super().done(result)

def _poll_fetcher(self) -> None:
    if self._closed:  # Guard against post-close polling
        return
```

### 1.4 Investigate idaapi.PluginForm crash — use `ida_kernwin.close_widget()`
When IDA Pro closes or the form is destroyed while `SettingsDialog` is open (as a modal), the Qt event loop may deliver destruction events to child widgets. In `done()` (settings_dialog.py:513), add:
```python
def done(self, result: int) -> None:
    self._closed = True
    self._init_timer.stop()
    self._poll_timer.stop()
    self._fetcher.shutdown()
    super().done(result)
```
The `_closed = True` flag should be set BEFORE calling parent `done()`. Also check if `_poll_timer` and `_init_timer` are properly parented to `self` (they are, via `QTimer(self)`). The problem might be that `_poll_timer` is started BEFORE `showEvent` (which is wrong — the timer should be started AFTER the dialog is shown).

### 1.5 `_poll_timer` started in `__init__` but fetcher not ready
`settings_dialog.py:221` starts `_poll_timer.start(150)` in `__init__`, but `_fetcher.fetch()` is only called in `_deferred_init()` (which runs AFTER `showEvent`). Until then, `_poll_fetcher()` polls an empty queue. This is fine but wasteful. More critically, the `_init_timer` is a single-shot with interval 0 — it fires at the NEXT event loop tick after `showEvent`. If the dialog is somehow closed before that tick (e.g. user presses Escape), `_deferred_init()` never runs but `_poll_timer` keeps polling forever.

**Fix**: Stop `_poll_timer` in `done()` (which already happens, line 517). Also set `_closed = True` before stopping timers to prevent `_poll_fetcher` from trying to access UI elements after the dialog is closed.

## Phase 2: Qt Performance Optimizations

### 2.1 PanelCore `_poll_events` — Adaptive singleShot chain (APPROVED)
Current: 50ms interval polls up to 20 events per tick. ~20 polls/sec when idle.

**Fix**: Replace fixed-interval `_poll_timer` with a `QTimer.singleShot` chain:
- When queue empty AND agent not running → schedule next poll at 200ms (5/sec idle)
- When queue empty AND agent IS running → schedule next poll at 100ms (10/sec)
- When events available → process up to 20, then reschedule immediately at 0ms (fast path during streaming)

Remove `_ensure_poll_timer()` / `_stop_poll_timer()` — replace with:
```python
def _schedule_poll(self, interval_ms: int) -> None:
    if self._is_shutdown:
        return
    QTimer.singleShot(interval_ms, self._poll_events)

def _poll_events(self) -> None:
    if self._polling or self._is_shutdown:
        return
    self._polling = True
    try:
        for _ in range(20):
            event = self._ctrl.get_event(timeout=0)
            if event is None:
                interval = 200 if not self._ctrl.is_agent_running else 100
                self._schedule_poll(interval)
                if not self._ctrl.is_agent_running:
                    self._on_agent_finished()
                return
            self._on_event(event)
    finally:
        self._polling = False
```
Start with `_schedule_poll(0)` when agent starts, stop by setting `_is_shutdown = True`.

### 2.2 `_poll_tools_events` — 100ms is too fast for subagent events
Subagent events are infrequent (spawn, complete, fail). 100ms = 10 polls/sec is wasteful.

**Fix**: Increase to `500ms`. Subagent tree updates don't need to be real-time.

### 2.3 Bulk renamer `_renamer_fetch_timer` — use `QTimer.singleShot` chain instead of continuous timer
The `_fetch_renamer_page` runs with `setInterval(0)` but schedules itself repeatedly via the function body. This is actually a self-scheduling chain which is better than a fixed interval — but the timer is never properly cleaned up if `_load_renamer_functions` is called twice.

**Fix**: Use `QTimer.singleShot(0, self._fetch_renamer_page)` chain and track `_renamer_fetch_timer = None` when done. Also check `hasattr(self, "_bulk_renamer")` at the START of `_load_renamer_functions` to prevent double-init.

### 2.4 `SettingsDialog._poll_timer` — 150ms is wasteful
`_poll_fetcher()` does a non-blocking `queue.get_nowait()` — almost always empty. 150ms = ~7 polls/sec for no data.

**Fix**: Increase to `500ms`.

### 2.5 `ChatView._scroll_timer` — 80ms coalescing is good, but `setSingleShot(True)` is correct
Already optimal — single-shot timer reset on each scroll request. No change needed.

### 2.6 Skills refresh timer — remove timer entirely, use signal
`PanelCore._ensure_skills_refresh_timer()` creates a 300ms polling timer to check if `skill_slugs` is populated.

**Fix**: Replace with a one-shot in `_refresh_skill_slugs` — if slugs not ready after first check, schedule a `QTimer.singleShot(300, self._refresh_skill_slugs)` instead of a continuous timer. Stop timer as soon as slugs arrive.

### 2.7 Lambda signal connections — potential reference capture
`panel_core.py` line 520: `chat_view.orchestra_approval_decided.connect(lambda ...)` — lambda captures `self` and local variables. PySide6 can hold references to deleted objects through lambdas, causing crashes.

**Fix**: Use `functools.partial` or bound methods:
```python
# Instead of lambda in _create_tab:
chat_view.orchestra_approval_decided.connect(self._on_orchestra_approval)
```

### 2.8 `RikuganPanel.on_database_changed` — shutdown/start race
When the database changes, `on_database_changed()` calls `shutdown()` then `OnCreate()` reinitializes. If an agent is running, this creates a race between the shutdown thread and the agent loop.

**Fix**: In `on_database_changed`, check `self._core._ctrl.is_agent_running` and if true, prompt user or queue the restart.

## Phase 6: Memory & Threading Fixes

### 6.1 `config.save()` — atomic write (APPROVED)
Current: direct write to config file. If process crashes mid-write, config is corrupted.

**Fix**: Write to `config.json.tmp` then `os.replace()` to target path (atomic rename):
```python
def save(self, password: str = "") -> None:
    path = self.config_path
    tmp = path + ".tmp"
    with open(tmp, "w") as f:
        json.dump(asdict(self._to_dict()), f, indent=2)
    os.replace(tmp, path)  # atomic on POSIX, decent on Windows
```

### 6.2 SubagentManager thread safety — verify lock coverage
Check all public methods of `SubagentManager` hold `self._lock`. The `register()`, `update_external()`, `cancel()`, `complete()` methods should all be locked.

### 6.3 `BulkRenamerEngine` — cancel check missing in `_fetch_renamer_page`
`_fetch_renamer_page` can keep running after cancel since it has no `_cancel.is_set()` check.

**Fix**: Add at the start of `_fetch_renamer_page`:
```python
if getattr(self, "_cancel", None) and self._cancel.is_set():
    self._renamer_fetch_timer.stop()
    self._renamer_fetch_timer.deleteLater()
    self._renamer_fetch_timer = None
    return
```

### 6.4 Skills refresh timer — convert to one-shot chain
Replace continuous 300ms polling with `QTimer.singleShot` chain. The check `if slugs` fails → reschedule at 300ms → stop when ready.
