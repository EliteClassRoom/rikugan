## 1. Research Summary

### 1.1 Skill System Overview
Skills in Rikugan are Markdown files with YAML frontmatter, located in:
- Built-in skills: `rikugan/skills/builtins/<slug>/SKILL.md`
- User skills: `~/.idapro/rikugan/skills/<slug>/SKILL.md`

**Frontmatter fields:**
- `name` - Display name
- `description` - One-line description
- `tags` - Categorization
- `triggers` - Keywords for auto-activation
- `mode` - Special modes (plan/exploration/empty)
- `allowed_tools` - Tool restrictions
- `author`, `version` - Metadata

**Skill body** is wrapped in `<skill name="...">` tags and sanitized before injection into the prompt.

**References directory** (`references/`) - Optional `.md` files that are auto-appended to the skill body.

# How to Search IDA Documentation Effectively

## Identifying What to Search For

### When User Asks About...
- "How do I read bytes at an address?" → ida_bytes module OR Domain API Bytes
- "How do I iterate functions?" → idautils.Functions() OR Domain API db.functions
- "How do I rename something?" → ida_name or idc OR Domain API db.names
- "How do I create a custom UI?" → ida_kernwin, UI examples
- "How do I work with structures?" → ida_typeinf (IDA 9+), Type examples
- "How do I use the decompiler?" → ida_hexrays, Decompilation examples
- "How do I add a breakpoint?" → ida_dbg, Debugger examples
- "How do I analyze xrefs?" → Domain API db.xrefs (simplest for IDA 9.1+)
- "How do I find strings?" → Domain API db.strings (simplest for IDA 9.1+)
- "What changed in IDA 9.2?" → Release notes for 9.2
- "How do I install domain API?" → Domain API Getting Started

## Decision: Domain API vs IDAPython

**Use Domain API when:**
- Task is straightforward (iterate functions, get xrefs, read bytes)
- Writing new scripts for IDA 9.1+
- Simpler, more readable code is preferred
- No need for advanced/specialized features

**Use IDAPython when:**
- Need advanced/deprecated features
- Working with version 8.x compatibility
- Specific SDK functionality required
- Contributing to existing codebase

# Example Queries and Expected Documentation

## Query: "How do I read a byte at an address?"

**Preferred (Domain API - IDA 9.1+):**
- Documentation: https://ida-domain.docs.hex-rays.com/ref/bytes/
- Code:
```python
from ida_domain import Database
with Database.open() as db:
    byte_val = db.bytes.get_byte_at(0x401000)
```

**Alternative (IDAPython):**
- Documentation: https://python.docs.hex-rays.com/ida_bytes/index.html#ida_bytes.get_byte
- Code:
```python
byte_value = idc.get_wide_byte(0x401000)  # Read a byte
```
**Module:** idc or ida_bytes
**Function:** get_byte, get_wide_byte

---

## Query: "How do I iterate all functions in the database?"

**Preferred (Domain API - IDA 9.1+):**
- Documentation: https://ida-domain.docs.hex-rays.com/ref/functions/
- Examples: https://ida-domain.docs.hex-rays.com/examples/#function-analysis
- Code:
```python
from ida_domain import Database
with Database.open() as db:
    for func in db.functions:
        print(f"{func.name}: {hex(func.start_ea)}")
```

**Alternative (IDAPython):**
- Documentation: https://python.docs.hex-rays.com/idautils/index.html#idautils.Functions
- Code:
```python
for func_ea in idautils.Functions():
    func_name = idc.get_func_name(func_ea)
    print(f"{hex(func_ea)}: {func_name}")
```
**Module:** idautils
**Function:** Functions()

---

## Query: "How do I analyze cross-references?"

**Preferred (Domain API - IDA 9.1+):**
- Documentation: https://ida-domain.docs.hex-rays.com/ref/xrefs/
- Examples: https://ida-domain.docs.hex-rays.com/examples/#cross-reference-analysis
- Code:
```python
from ida_domain import Database
with Database.open() as db:
    for xref in db.xrefs.to_ea(target_addr):
        print(f"From {hex(xref.from_ea)} to {hex(xref.to_ea)}")
```

**Alternative (IDAPython):**
- Documentation: https://python.docs.hex-rays.com/idautils/index.html#idautils.XrefsTo
- Code:
```python
for xref in idautils.XrefsTo(target_addr):
    print(f"From {hex(xref.frm)} to {hex(xref.to)}")
```

---

# Plan: AOrchestra-Style Agent Orchestration for Rikugan

## 1. Overview

### 1.1 What is AOrchestra?
AOrchestra (from FoundationAgents) implements **automated sub-agent creation** for agentic orchestration. It models any sub-agent as a **four-tuple interface φ = ⟨I, C, T, M⟩**:

- **I (Instruction)**: Actionable instructions for the sub-agent
- **C (Context)**: Curated context from the main agent
- **T (Tools)**: Selected tools available to the sub-agent
- **M (Model)**: Chosen model for the sub-agent

The **MainAgent** orchestrates by:
1. Decomposing the overall task into subtasks
2. Dynamically creating specialized sub-agents via the four-tuple
3. Managing context sharing between sub-agents
4. Handling completion via tools like `delegate_task`, `submit`, `complete`

### 1.2 Rikugan's Existing Subagent System
Rikugan already has:
- `SubagentRunner` - Isolated agent runner
- `SubagentManager` - Registry/orchestrator for subagents
- `spawn_subagent` pseudo-tool
- `Perks` system for specialization

**Gap**: Rikugan's subagents are:
- Created with fixed configurations (model, tools)
- Lack dynamic specialization per subtask
- No `delegate_task`/`submit`/`complete` orchestration tools
- No configurable context passing

### 1.3 Implementation Goal
Extend Rikugan's existing `SubagentManager` to support AOrchestra-style orchestration:
- MainAgent with orchestration tools
- Dynamic sub-agent specialization (four-tuple φ)
- Configurable context passing
- Multiple model selection per sub-agent
- Task completion management

### 1.4 Design Decisions (Confirmed)
- **Mode**: Opt-in alternative. OrchestraMainAgent is activated via `/orchestra` command. Existing agent loop remains unchanged.
- **Approval**: All `delegate_task` calls require user approval (Approve/Reject buttons). Never auto-execute.
- **Scope**: Full 4-phase implementation including OrchestraPanel UI.

## 3. Tool Design

### 3.1 `delegate_task` Tool
```python
@tool(category="orchestration")
def delegate_task(
    instruction: Annotated[str, "Detailed instruction for the sub-agent"],
    context: Annotated[str, "Relevant context from the main task"],
    tools: Annotated[list[str], "Tools to make available to sub-agent"],
    model: Annotated[str, "Model to use (from sub_models list)"],
    max_steps: Annotated[int, "Max steps for sub-agent"] = 30,
) -> str:
    """
    Delegate a subtask to a specialized sub-agent.
    
    The sub-agent will be created with the four-tuple φ = <I, C, T, M>:
    - I: Your instruction
    - C: Context you provide
    - T: Tools you specify
    - M: Model you select
    
    Returns a summary of the sub-agent's findings.
    """
```

**Response format:**
```json
{
  "status": "done" | "partial",
  "result": "The sub-agent's findings",
  "completed": ["list of completed items"],
  "issues": ["list of remaining issues"],
  "message": "Status message",
  "trace_summary": "Brief trace of actions taken"
}
```

### 3.2 `submit` Tool
```python
@tool(category="orchestration")  
def submit(
    reason: Annotated[str, "Final reasoning/submission message"],
    result: Annotated[str, "The final answer or result"] = "",
) -> str:
    """Submit the final result of the orchestration task."""
```

### 3.3 `complete` Tool
```python
@tool(category="orchestration")
def complete(
    answer: Annotated[str, "The final answer"],
) -> str:
    """Mark the task as complete with the given answer (for GAIA-style tasks)."""
```

## 9. Security Considerations

### 9.1 Sub-agent Permissions
- Sub-agents should only have tools explicitly passed via `delegate_task`

### 9.2 Context Sanitization
- Sanitize context before passing to sub-agent
- Strip any injection markers
- Respect token limits

### 9.3 Resource Limits
- Max delegations per task
- Max sub-agent steps
- Max context window
- Timeout per sub-agent

### 9.4 Approval Gates
- `delegate_task` always requires approval (like `execute_python`)
- `submit`/`complete` may require approval based on config
